# 🍪 쿠키 기반 인증 구현 시도 및 기술 분석 보고서

본 문서는 잡코리아 자동화 봇의 세션 유지를 위해 수행한 쿠키 주입(Cookie Injection) 과정과 현재의 기술적 한계를 상세히 기록합니다.

## 1. 쿠키 구현 전략 및 노력 (Efforts)

잡코리아의 강력한 보안 시스템(2단계 인증, 캡차)을 우회하기 위해 다음과 같은 단계적 전략을 시행하였습니다.

### 🛠️ 쿠키 캡처 스크립트 (`save_cookies.py`) 개발
* **1차 시도 (자동화)**: 아이디와 비밀번호를 자동으로 입력하고 2단계 인증 단계에서만 사용자가 개입하여 쿠키를 `.pkl` 파일로 저장하는 방식을 구현하였습니다.
* **2차 시도 (수동 안전 모드)**: 보안 탐지를 최소화하기 위해 브라우저 실행 후 사용자가 직접 로그인 및 대상 페이지 방문까지 완료한 뒤 세션 전체를 캡처하는 '인터랙티브 모드'로 전환하였습니다.
* [cite_start]**세션 예열 (Session Warming)**: 단순히 로그인 직후의 쿠키가 아니라, 실제 데이터가 있는 `ACCEPT_URL`을 한 번 방문하여 권한이 완전히 승인된 상태의 쿠키를 획득하도록 프로세스를 개선하였습니다[cite: 1].

### 🛠️ 쿠키 주입 및 검증 로직 (`driver.py`) 고도화
* **쿠키 주입**: 저장된 `.pkl` 파일을 읽어 `driver.add_cookie()`를 통해 셀레니움 브라우저에 이식하는 기능을 구현하였습니다.
* **검증 로직 강화**: 단순히 메인 페이지 접속 여부를 확인하던 기존 방식에서, 실제 권한이 필요한 `ACCEPT_URL`에 접속한 뒤 리다이렉트(Redirect) 여부를 체크하여 '최종 권한 획득 성공'을 판단하도록 코드를 고도화하였습니다.

## 2. 현재 상황 및 실패 분석 (Current Status & Failure Analysis)

현재 봇은 쿠키를 통해 로그인을 성공적으로 수행하는 듯하나, 실제 데이터 수집 단계에서 차단되는 상태입니다.

### 🛑 주요 현상 (Symptom)
* **로그 상의 성공**: `[로그인] ✅ 최종 권한 획득 성공!` 메시지 출력.
* **수집 단계의 실패**: 1페이지 이동 시 페이지 번호를 찾지 못하고 종료됨.
* **리다이렉트 발생**: 디버깅 스냅샷 확인 결과, 봇이 위치한 현재 URL이 **2단계 인증(TwoFactorAuth)** 페이지로 강제 이동됨을 확인하였습니다.
    * **접속 URL**: `https://www.jobkorea.co.kr/user/Certify/TwoFactorAuth?reUrl=...`

### 🔍 기술적 원인 분석 (Root Cause)
1.  **IP 불일치에 따른 세션 무효화**: 로컬 PC(윈도우)에서 생성한 쿠키를 서버(오라클 클라우드) IP에서 사용할 때, 잡코리아 서버가 '접속 환경 변경'을 감지하고 추가 기기 인증을 요구하고 있습니다.
2.  **Headless 패턴 감지**: 서버의 화면 없는 브라우저 특성(Headless)을 잡코리아 보안 솔루션이 감지하여, 쿠키가 있더라도 사람이 아님을 의심하고 다시 인증을 요구하는 것으로 판단됩니다.
3.  **세션 보안 정책**: 잡코리아의 세션 보안 주기가 매우 짧거나, 특정 페이지(후보자 목록 등) 접근 시마다 서버 사이드에서 실시간 권한 재검증을 수행하고 있습니다.

## 3. 향후 개선 방향
* **서버 직접 인증**: 서버 환경에서 직접 2단계 인증 번호를 입력받아 세션을 생성하는 방식 검토.
* **브라우저 핑거프린트 우회**: Headless 브라우저를 실제 브라우저처럼 보이게 하는 `stealth` 옵션 강화.
* **고정 IP 확보**: 신뢰할 수 있는 고정 IP 환경에서의 세션 유지 시도.
